# Heimdell Verified Consent API — Database Setup Guide

This guide covers connecting a PostgreSQL database so the API routes work end-to-end.

> **Security reminder:** Never paste your `DATABASE_URL`, passwords, or API keys into chat, commits, screenshots, pull requests, or documentation. These values belong only in `.env.local` on your local machine, or in your hosting provider's secret manager.

---

## Recommended option — Supabase PostgreSQL (free tier available)

### 1. Create a Supabase project

1. Go to [https://supabase.com](https://supabase.com) and sign in.
2. Click **New project**.
3. Choose a name, region, and a strong database password. **Save the password immediately** — you will need it in a moment.
4. Wait for the project to finish provisioning (30–60 seconds).

### 2. Get the connection string

1. In the left sidebar, click **Project Settings**.
2. Click **Database**.
3. Scroll down to **Connection string**.
4. Select the **URI** tab.
5. If you are on a Supabase plan that supports connection pooling, prefer the **Session mode** or **Transaction mode** pooler connection string — these are more reliable for serverless environments.
6. For Netlify/serverless deployments, add a conservative Prisma connection limit to the pooled URL, such as `connection_limit=1` or `connection_limit=2`, depending on the Supabase pool size available to the project.
7. Copy the full connection string. It looks like:

```
postgresql://postgres.[project-ref]:[YOUR-PASSWORD]@aws-0-eu-west-2.pooler.supabase.com:5432/postgres
```

8. Replace `[YOUR-PASSWORD]` with your actual database password.

> **Note:** If your password contains special characters (such as `@`, `#`, `!`, `%`), they must be URL-encoded. For example `p@ss!` becomes `p%40ss%21`. Use a URL encoder if unsure.

### 3. Paste into `.env.local`

Open `.env.local` in the project root and replace the placeholder:

```env
DATABASE_URL="postgresql://postgres.[project-ref]:your-password@aws-0-eu-west-2.pooler.supabase.com:5432/postgres?sslmode=require&connection_limit=1"
APP_URL="http://localhost:3000"
ENCRYPTION_KEY="<generate-with-npm-run-generate:encryption-key>"
```

Save the file. Do not commit it — `.env.local` is in `.gitignore`.

## Alternative option — Neon PostgreSQL (serverless, free tier available)

1. Go to [https://neon.tech](https://neon.tech) and sign in.
2. Click **New project**.
3. Choose a name and region.
4. Click **Dashboard** → **Connection Details**.
5. Copy the **Connection string** (it begins with `postgresql://`).
6. Paste it into `.env.local` as `DATABASE_URL`.

> Neon connection strings do not require URL-encoding in most cases, but verify your password does not contain `@` or `#`.

## Local option — Docker Postgres

If you prefer a fully local database:

```bash
docker run --name hvcs-db -e POSTGRES_PASSWORD=devpassword -e POSTGRES_DB=heimdell_consent -p 5432:5432 -d postgres:16
```

Then set in `.env.local`:

```env
DATABASE_URL="postgresql://postgres:devpassword@localhost:5432/heimdell_consent"
```

## Full `.env.local` reference

```env
# Required — replace with real connection string before any db:* commands
DATABASE_URL="postgresql://user:password@host:5432/dbname"

# Required — base URL for verification links sent to customers
APP_URL="http://localhost:3000"

# Required — generate once with: npm run generate:encryption-key
# Must be a base64-encoded 32-byte key (256-bit key)
ENCRYPTION_KEY="your-base64-encoded-32-byte-key"

# Required for dashboard auth
NEXT_PUBLIC_SUPABASE_URL="https://your-project-ref.supabase.co"
NEXT_PUBLIC_SUPABASE_ANON_KEY="your-supabase-anon-key"

# Optional — populated automatically after running npm run seed:dev-client
# For local shell testing only. Never commit this.
DEV_API_KEY="hvcs_dev_..."
```

## Required setup commands

Run these **in order** after filling in `.env.local`:

```bash
# 1. Generate a secure encryption key (if not already done)
npm run generate:encryption-key

# 2. Validate the Prisma schema
npm run db:validate

# 3. Apply migrations to a development database
npm run db:migrate:dev

# 4. Regenerate the Prisma client from the schema
npm run db:generate

# 5. Seed a development API client (prints a raw API key once)
npm run seed:dev-client

# 6. Start the dev server
npm run dev
```

All `db:*` scripts automatically load `.env.local` via `dotenv-cli` — you do not need to export `DATABASE_URL` manually.

## Encryption key requirements

`ENCRYPTION_KEY` is required before any sale intake request can store Direct Debit account details. It must be a base64-encoded 32-byte key generated by:

```bash
npm run generate:encryption-key
```

The application validates this key before encrypting or decrypting sensitive values. If the key is missing, malformed, or decodes to anything other than exactly 32 bytes, sale intake fails with a safe server configuration error. Do not reuse placeholder values.

Encrypted account numbers are stored with AES-256-GCM as `v1:<iv>:<auth_tag>:<ciphertext>`. Existing local development rows created with the old base64 placeholder format cannot be safely migrated by the app; reset development data if those rows are no longer readable.

## Supabase Auth setup

Dashboard authentication uses Supabase Auth server-side sessions. Required env vars:

```env
NEXT_PUBLIC_SUPABASE_URL="https://your-project-ref.supabase.co"
NEXT_PUBLIC_SUPABASE_ANON_KEY="your-supabase-anon-key"
```

Use the anon key only. Do not put the Supabase service-role key in frontend code or any `NEXT_PUBLIC_*` variable.

Recommended setup:

1. Configure Supabase Auth email magic links.
2. Keep public self-serve signup disabled for dashboard users unless explicitly approved.
3. Invite or admin-create the dashboard user in Supabase.
4. Apply migrations, including the `User.externalAuthId` migration.
5. Create a matching internal `User` row where `externalAuthId` is the Supabase auth user ID.
6. Create an `OrganizationMembership` row for that user.

Dashboard pages remain placeholder-only until the live dashboard data phase.

Use the safe setup helper instead of manual SQL for local/admin setup:

```bash
npm run setup:dashboard-user -- \
  --org-name "Acme Broadband Ltd" \
  --org-slug "acme-broadband" \
  --email "admin@example.com" \
  --external-auth-id "supabase-user-uuid" \
  --role OWNER
```

See [DASHBOARD_SETUP.md](DASHBOARD_SETUP.md) for role assignments and optional development client linking.

## After seeding — save your API key

`npm run seed:dev-client` prints a raw API key **once only**. It is not stored in plain text anywhere in the system.

**Copy it immediately** and store it somewhere safe (password manager recommended for real use).

For local shell testing, save it as an environment variable in your current terminal session:

**macOS / Linux / Git Bash:**
```bash
export HVCS_API_KEY="hvcs_dev_..."
```

**Windows PowerShell:**
```powershell
$env:HVCS_API_KEY="hvcs_dev_..."
```

Use this key as the `x-api-key` header on all authenticated API requests. See [API_TESTING.md](API_TESTING.md) for full curl examples.

> **Production note:** Do not store API keys in plain text in production. Use your hosting provider's secret manager (e.g. Supabase Vault, Vercel Environment Variables, AWS Secrets Manager).

## Common errors

| Error | Cause | Fix |
|---|---|---|
| `P1001: Can't reach database server` | Wrong host, port, or firewall | Verify the host and port in your connection string. Check Supabase/Neon firewall rules. |
| `P1000: Authentication failed` | Wrong password | Re-copy the password. Check for special characters that need URL-encoding (see above). |
| `relation "..." does not exist` | Migrations not applied | Run `npm run db:migrate:dev` for local dev or `npm run db:migrate:deploy` for reviewed deployments. |
| `invalid connection string` | Malformed `DATABASE_URL` | Ensure the string starts with `postgresql://` and has no unescaped special characters. |
| `DATABASE_URL not found` | `.env.local` missing or wrong key name | Check `.env.local` exists at the project root and the key is spelled `DATABASE_URL` exactly. |
| `EMAXCONNSESSION` / max clients reached | Serverless functions opened too many pooled sessions | Use the Supabase pooler URL and add a low Prisma `connection_limit`, such as `connection_limit=1`, then redeploy. |
| SSL error / certificate verification | Supabase requires SSL | Append `?sslmode=require` to your connection string if not already present. |
| `SCRAM authentication requires a non-empty password` | Password is empty | Set a real password on the Supabase project and update `.env.local`. |
| `Error: secretOrKey must have a value` | Unrelated env var missing | Check all required `.env.local` keys are populated. |

## What not to do

- **Do not commit `.env.local`.** It is in `.gitignore` — keep it that way.
- **Do not use the Supabase service role key as your database password.** The service role key is a JWT for the Supabase REST API, not a Postgres credential.
- **Do not paste secrets into source code**, comments, or markdown files in the repository.
- **Do not run `npm run db:push` against a production database.** `db:push` bypasses migration history and can make drift unsafe. Use `npm run db:migrate:deploy` for production after backup and review.
- **Do not expose the Supabase database directly to the internet.** Use Supabase's built-in connection pooler and restrict IP ranges in the Supabase dashboard where possible.
- **Do not share your Supabase project reference or connection string** in support tickets, screenshots, or public repositories.
